## Reflection

- Object as a human metaphor was quite a revelation.
- I would surely prefer metaphor over concrete code. Now, regarding object thinking vs procedural thinking. I like the idea that everything is an object... But let's not forget that everything is a function as well. Let's look at Church's lambda expressions - it feels almost like we don't need objects or data at all if we have enough compute power. At certain level everything becomes functions just like in the world of physics with a certain precision everything becomes a wave. Yet at another angle - I also love the Unix idea of everything is a text. So. I probably don't like procedural habits. But whether it's everyhing-is-an-object vs everything-is-a-function vs everything-is-a-text I have not decided yet.
- Null object vs null references/exceptions... Actually why not Monads an Options pattern? It seems like this is a direction C# is heading to and for a good reason.
- Philosophical depth vs technical pragmatism - Is there really any difference between these two nowadays?
- Some other ideas I noted down while listening:
	- Systems as communities. Software as human
	- Emergent behavior
	- Distributing responsibilities to autonomous actors (Actor Model?)
	- West is making references to Christopher Alexander's Timeless Way of Building and how it goes deeper than a most popular Pattern Language. I would like to explore more about how both of these books influenced software movement (maybe even beyond GoF)
	- Good systems fail fast and loud - I think I'll stick to this notion at my job.
	- Moving from conducted orchestra to a jazz band produces a fear of cacophony.
	- What if every object instantiated in my C# code becomes an autonomous intelligent Agent
